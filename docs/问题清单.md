## 一、已确认存在的问题

### P0 安全与成本风险

#### 1. 前端 API Token 编译进 JS Bundle ✅ 已修复
- 位置：~~`frontend/src/api/reviewApi.ts`~~ → 已迁移至 `frontend/vite.config.ts`
- 描述：~~`VITE_REVIEW_API_TOKEN` 通过 `import.meta.env` 读取后，会被 Vite 在构建时直接内联到 JS 产物中。~~
- 状态：已修复 — 于 2026-05-29 将 Token 从前端代码移除，改为由 Vite 开发代理（`/api` proxy）服务端注入 `Authorization` header。前端不再接触 Token。生产环境由反向代理（Nginx）注入。

### P1 可靠性与容错

#### 2. 速率限制器无法跨 Worker 工作
- 位置：`backend/app/core/rate_limit.py`
- 描述：限流器使用进程内 `defaultdict(deque)` + `threading.Lock` 存储计数。当使用 `uvicorn --workers N`（N > 1）部署时，每个 Worker 拥有独立的内存空间和独立的计数器，限流形同虚设。服务重启后计数也会归零。
- 影响：多 Worker 部署下限流机制完全失效，无法防御批量请求。
- 建议：MVP 阶段可单 Worker 部署（`--workers 1`）；生产环境改用 Redis 滑动窗口或将限流移至 Nginx/网关层。

### P2 性能与分析逻辑

#### 3. LLM 调用缺少重试机制 ✅ 已修复
- 位置：`backend/app/services/llm_review.py`
- 描述：LLM API 的瞬时故障（超时、限流、服务不可用、连接失败）会直接导致请求返回 502，无自动重试。
- 影响：LLM API 的不稳定性较常见，缺少重试会降低服务可用性。
- 状态：已修复 — 于 2026-05-29 添加最多 3 次指数退避重试（1s → 2s → 4s），仅对可恢复错误类型重试。

#### 4. GitHub 文件分页没有最大上限
- 位置：`backend/app/services/github_pr.py`
- 描述：分页使用 `while True`，只有当返回数量小于 100 时才退出。
- 影响：理论上在异常条件下可能产生过多外部请求。
- 建议：增加最大页数保护并记录截断日志。

#### 5. `ReviewState.errors` 当前是死字段
- 位置：`backend/app/agents/state.py`
- 描述：state 中定义了 `errors`，但工作流节点没有读写它。
- 影响：增加理解负担，容易让人误以为存在统一错误收集机制。
- 建议：删除该字段或真正实现错误收集逻辑。

#### 6. 分页数据中非 dict 元素被静默丢弃
- 位置：`backend/app/services/github_pr.py`
- 描述：`_get_paginated_json` 用列表推导式过滤非 dict 元素：`page_items = [item for item in data if isinstance(item, dict)]`。如果某页所有元素均非 dict（低概率但可能），`len(page_items) < 100` 直接退出，返回不完整数据，无日志警告。
- 影响：数据完整性风险，排查困难。
- 建议：当前页非 dict 元素数量大于 0 时记录警告日志；当 `len(data) > 0` 但 `len(page_items) == 0` 时向上抛异常而非静默返回。

#### 7. 上下文文件选择无优先级排序
- 位置：`backend/app/agents/nodes.py`
- 描述：`_select_files_for_context` 只按顺序取前 30 个未跳过的文件，不做任何优先级排序。当 PR 有 50+ 代码文件时，核心业务逻辑文件可能被排在前面的小配置文件挤出。
- 影响：大 PR 的分析质量下降，LLM 看不到关键变更。
- 建议：按文件改动行数降序排列后再截取，确保大改动文件优先入模。

#### 8. 风险项不去重
- 位置：`backend/app/agents/nodes.py`
- 描述：`_normalize_analysis` 对 suggestions 做了去重（by file+type+comment），但对 risks 未做任何去重处理。LLM 经常将同一问题用不同措辞输出多次。
- 影响：报告中出现重复风险项，降低报告可读性和信任度。
- 建议：按 (file, category, issue) 对 risks 去重，保留 confidence 最高的版本。

#### 9. `ReviewState` 使用 `total=False` 缺少运行时安全保障
- 位置：`backend/app/agents/state.py`
- 描述：`ReviewState` 定义为 `TypedDict(total=False)`，所有字段均为可选。如果某个节点未能正确填充其输出字段，下一个节点在访问 `state["pr_data"]` 时会抛出 `KeyError`，最终呈现为无上下文的 500 错误。
- 影响：节点间的数据契约不清晰，排查问题困难。
- 建议：将 `total=False` 改为 `total=True`，在 initial state 中使用 `typing.cast` 或显式声明每个节点所需的输入字段。

### P3 工程化缺口

#### 10. 缺少自动化测试
- 位置：全局
- 描述：当前没有后端 `pytest`、前端单元测试或端到端测试。
- 影响：后续改动容易引入回归，尤其是 GitHub API、LLM 输出校验和前端渲染链路。
- 建议：优先补后端 API 测试和关键服务单元测试。

#### 11. 缺少 lint / format / CI
- 位置：全局
- 描述：当前没有 `ruff`、`mypy`、`eslint`、`prettier`、GitHub Actions 等工程化约束。
- 影响：代码风格、类型问题和构建问题只能靠人工发现。
- 建议：逐步补齐最小工具链：后端 `ruff`，前端 `eslint`，以及基础 CI。

#### 12. VSCode Python 环境配置与 README 不一致 — 已不适用
- 位置：~~`.vscode/settings.json`~~、`Readme.md`
- 状态：`.vscode/` 目录已在 commit `4f4904f` 中删除，此问题不再适用。

#### 13. 缺少 `.env.example` 模板文件 — 已不适用
- 位置：`backend/`、`frontend/`
- 状态：`backend/.env.example` 和 `frontend/.env.example` 已存在，此问题不再适用。

#### 14. 依赖缺少 lock 文件
- 位置：`backend/requirements.txt`、`frontend/package.json`
- 描述：后端使用精确版本固定（`==`）但没有 lock 文件（如 `requirements.lock` 或 `uv.lock`），前端 `package.json` 使用 `^` 范围版本。两者的依赖树均无法保证跨环境完全可复现。
- 影响：不同时间安装依赖可能得到不同的子依赖版本，导致难以排查的环境差异问题。
- 建议：后端生成 lock 文件（`pip freeze > requirements.lock`），前端提交 `package-lock.json`。

## 三、补充发现的问题

### 1. 文档与实际实现存在轻微偏差
- 位置：`docs/设计文档.md`、`docs/任务书.md`、当前代码
- 描述：部分文档仍描述了用户系统、MySQL、Redis、Docker Compose、多页面前端等未来态能力，而当前实现仍是 MVP 版。
- 影响：读文档的人容易误判当前交付范围。
- 建议：在设计文档中显式区分“已实现”和“规划中”。

### 2. 前端默认 PR URL 为固定公开仓库
- 位置：`frontend/src/App.vue`
- 描述：输入框默认值写死为 `https://github.com/fastapi/fastapi/pull/1`。
- 影响：便于演示，但如果不说明，会让人误以为这是系统要求格式示例而非默认演示数据。
- 建议：保留演示值可以，但建议加更清晰的 placeholder 或说明。

### 3. Clipboard 调用未做异常兜底
- 位置：`frontend/src/App.vue`
- 描述：调用 `navigator.clipboard.writeText()` 时没有 catch。
- 影响：在非安全上下文、旧浏览器或权限受限环境下，复制失败会直接抛异常。
- 建议：增加失败提示并优雅降级。

## 五、当前状态总结

截至 2026-05-29，项目已修复 4 个问题（业务接口认证、限流保护、LLM 调用重试、前端 Token 泄露），3 个条目确认不再适用（VSCode 配置、缺 .env.example、.env 跟踪），新识别出 6 个待处理问题（速率限制器跨 Worker 失效、分页数据静默丢弃、文件选择无优先级、风险项不去重、ReviewState 运行时安全、缺 lock 文件）。
