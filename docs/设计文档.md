# AI Code Review — 系统设计文档

## 一、产品概述

AI Code Review 是一款面向开发者的 Pull Request 智能评审辅助工具。用户指定 GitHub PR 链接，系统自动获取代码变更，通过大语言模型进行多维度分析，生成结构化的评审报告，帮助 Reviewer 快速定位风险、提升评审效率与质量。

### 1.1 核心价值

- **提效**：将 Reviewer 从逐行通读 diff 中解放，先看 AI 标记的高风险区域
- **降漏**：通过规则化检查清单覆盖常见漏洞模式，减少人为疏忽
- **标准化**：统一评审维度（安全、性能、可维护性、测试覆盖等），消除 Reviewer 经验差异

### 1.2 使用场景

- Reviewer 打开 PR 后，先查看 AI 评审报告了解全局变更和风险点
- 新成员通过 AI 建议学习团队的代码评审标准
- 大 PR（50+ 文件）快速定位高风险区域，优先审查

### 1.3 定位原则

- **辅助而非替代**：报告是 Reviewer 的参考意见，不做硬性门禁
- **通用工具**：不限定语言或框架，支持主流后端与前端语言
- **开箱即用**：用户配置 GitHub Token 和 LLM API Key 后即可使用

### 2.2 架构决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 后端框架 | FastAPI | 异步原生支持，适合 I/O 密集型分析任务；Pydantic 强类型校验；OpenAPI 自动文档 |
| 前端框架 | Vue 3 + Vite | 响应式数据流适合报告展示；Vite 构建快；生态成熟 |
| AI 编排 | LangGraph | 有向图编排优于手动串行，节点可独立测试、可并行化扩展 |

| 缓存 | Redis 7 | 会话管理 + 分析结果缓存（同一 PR 短期内不重复分析） |
| 部署 | Docker Compose | 一键启动全栈，适合中小团队自托管 |

### 4.2 关键表结构

**users** — 用户表
```sql
CREATE TABLE users (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    email       VARCHAR(255) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,         -- bcrypt hash
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**tokens** — Token 表
```sql
CREATE TABLE tokens (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    user_id     INT NOT NULL,
    token_enc   TEXT NOT NULL,                 -- AES-256-CBC 加密存储
    is_valid    BOOLEAN DEFAULT TRUE,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**review_records** — 评审记录表
```sql
CREATE TABLE review_records (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    user_id         INT NOT NULL,
    pr_url          VARCHAR(512) NOT NULL,
    pr_title        VARCHAR(512),
    owner           VARCHAR(128),
    repo            VARCHAR(128),
    pr_number       INT,
    status          ENUM('pending','running','completed','failed') DEFAULT 'pending',
    summary_json    JSON,                      -- ReviewSummary 结构化存储
    result_json     JSON,                      -- 完整 ReviewResult
    file_count      INT DEFAULT 0,
    risk_counts     JSON,                      -- {high:N, medium:N, low:N}
    duration_ms     INT,
    pr_sha          VARCHAR(64),               -- 用于缓存去重
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    completed_at    DATETIME,
    FOREIGN KEY (user_id) REFERENCES users(id),
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
);
```

**feedback** — 反馈表
```sql
CREATE TABLE feedback (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    record_id   INT NOT NULL,
    risk_index  INT NOT NULL,                  -- 对应 result_json 中 risk 的索引
    rating      ENUM('helpful','not_helpful','false_positive') NOT NULL,
    comment     TEXT,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (record_id) REFERENCES review_records(id),
    INDEX idx_record_id (record_id)
);
```

### 4.3 Redis 数据结构

| Key Pattern | 类型 | 用途 | TTL |

## 六、模型选择与上下文策略

### 6.1 模型选择与接入设计

**接入方式：OpenAI 兼容接口**

用户配置 `OPENAI_API_KEY` 和 `OPENAI_BASE_URL`，系统通过同一套接口适配 OpenAI、DeepSeek、智谱 GLM、通义千问、Moonshot 等所有 OpenAI 兼容的 LLM 服务。这个设计的优势在于：

- **零厂商锁定**：不绑定任何单一模型厂商，用户可根据成本、速度、质量自由选择
- **渐进式升级**：新模型只要遵循 OpenAI 接口规范，无需改动代码即可接入
- **成本可控**：用户使用自己的 API Key，按量付费，系统无额外 LLM 成本

**模型推荐与场景匹配**

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 日常评审（快速） | DeepSeek-V3 / GPT-4o-mini | 成本低、速度快，适合常规代码审查 |
| 深度分析（准确） | Claude 4 Sonnet / GPT-4o | 长上下文理解强，适合大 PR 和复杂逻辑 |
| 安全审查 | Claude 4 Opus / GPT-4.5 | 最强的代码理解和漏洞识别能力 |

系统在创建评审时允许用户选择分析深度（快速/标准/深度），对应的模型配置可由用户自定义。

### 6.2 上下文获取与优化策略

**当前实现（Demo 级）**

- 仅取 diff patch（每个文件的增删行）
- 过滤锁文件、二进制、构建产物等无关文件
- 单次 Prompt 输入所有 diff，限制 6000 字符/文件、最多 30 文件

**生产级改进方向**

1. **目标文件完整上下文**：不仅取 diff，还取目标文件的完整内容（`GET /repos/{owner}/{repo}/contents/{path}?ref={head}`），让 LLM 看到完整的函数签名、变量定义和周围代码

2. **关联文件发现**：解析 diff 中 `import`/`require`/`include` 的变更，自动拉取被引用的文件，构建跨文件分析上下文

3. **智能截断**：不是简单的 6000 字符硬截断，而是按函数/类边界截断，确保提交的代码片段在语义上完整

4. **上下文优先级排序**：核心业务逻辑文件 > 工具函数 > 配置文件 > 文档，优先保证核心文件的完整上下文

5. **分层缓存**：PR diff 按 sha 缓存；文件内容按 blob sha 缓存；分析结果按（pr_sha + 文件 sha 集合）缓存

### 6.3 误报与漏报控制

| 阶段 | 策略 | 描述 |
|------|------|------|
| Prompt 层 | 结构化约束 | 要求 LLM 对每个风险项明确引用具体行号和代码片段，无法定位的视为低置信度 |
| Prompt 层 | 强制区分 | 要求区分"确定问题"和"建议关注"，后者标记为 low severity |
| 后处理 | 置信度阈值 | 过滤 confidence < 0.5 的风险项 |
| 后处理 | 去重合并 | 同一文件同一行的同类风险合并为一条 |
| 后处理 | 文件校验 | 删除引用了不存在的文件的风险项 |
| 反馈层 | 标记闭环 | 用户标记误报后，同类模式的风险在后续评审中自动降级或过滤 |
| 评估层 | 质量仪表盘 | 持续统计各风险类型、各语言的误报率，针对性优化提示词 |

## 七、Agent 工作流演进

### 7.1 当前：单 Agent 流水线（LangGraph）

```
parse_pr_url → fetch_pr_data → build_context → analyze_review → validate_result
```

五节点串行执行，state 在节点间传递。适合当前阶段的功能验证和 Demo 展示。

### 7.2 演进：多 Agent 并行（目标架构）

```
                      ┌──────────────────┐
                      │  Orchestrator    │
                      │  Agent           │
                      │  (任务分发/汇总)  │
                      └────────┬─────────┘
                               │
               ┌───────────────┼───────────────┐
               ▼               ▼               ▼
        ┌────────────┐  ┌────────────┐  ┌────────────┐
        │ Security    │  │ Performance│  │ Style &    │
        │ Reviewer    │  │ Reviewer   │  │ Maintain   │
        │ (安全审查)  │  │ (性能审查)  │  │ Reviewer   │
        └────────────┘  └────────────┘  └────────────┘
               │               │               │
               └───────────────┼───────────────┘
                               ▼
                      ┌──────────────────┐
                      │  Aggregator      │
                      │  (结果合并/去重)  │
                      └──────────────────┘
```

每个子 Agent 携带领域专用提示词和检查规则，独立分析同一份 diff。Orchestrator 负责任务分发，Aggregator 负责结果去重、冲突消解和评分归一化。

### 7.3 演进路线

| 阶段 | 时间 | 目标 | 关键变化 |
|------|------|------|----------|
| Stage 1 | 当前 | 单 Agent MVP | 流水线验证可行性 |
| Stage 2 | 生产层完成后 | 多 Agent 串行 | 增加安全/性能/风格三条专用 Agent，串行执行 |
| Stage 3 | 创新层 | 多 Agent 并行 | LangGraph 子图并行执行 + 结果聚合 |
| Stage 4 | 未来 | 知识增强 | 接入团队代码规范库、历史 Bug 模式库、CVE 漏洞库 |

- **数据持久化**：MySQL 数据卷挂载到宿主机，Redis RDB 持久化开启
- **日志收集**：所有容器日志输出到 stdout，通过 `docker logs` 统一查看
- **环境变量**：敏感配置（数据库密码、JWT 密钥、加密密钥）通过 `.env` 文件注入，不进入镜像
- **网络隔离**：MySQL 和 Redis 仅暴露给 backend 容器，不对外

## 十、设计思路总结

### 模型选择

采用 OpenAI 兼容接口作为统一接入层，避免厂商锁定。用户自备 API Key 按需选择模型，系统不做绑定。日常快速评审可用 DeepSeek-V3（成本低），重大 PR 深度分析可用 Claude 4 或 GPT-4o（准确度高），灵活性与成本可控兼得。

### 上下文获取

分三级递进：基础级（diff patch）→ 增强级（+ 目标文件全文 + 关联 import 文件）→ 感知级（+ 项目结构 + 历史 Bug 模式）。当前 Demo 为基础级，生产版至少达到增强级，创新版追求感知级。

### 未来扩展方向

1. **多 Agent 并行**：安全/性能/风格三条专用 Agent 并行分析，结果聚合
2. **知识增强**：接入 CVE 漏洞库、团队代码规范库、历史评审经验库
3. **IDE 插件**：将评审结果嵌入 VS Code / JetBrains，审查时无需切换界面
4. **CI 集成**：GitHub Actions / GitLab CI 中自动运行评审，结果直接评论到 PR
5. **团队空间**：多人协作的评审工作台，共享规则、共享评审记录、统计面板
